Confirmed via `grep -n "DEVELOPMENT_TEAM" project.pbxproj` (no matches) and `find . -iname "*.entitlements"` (no matches) that **this project has never had a Development Team configured** — every target uses `CODE_SIGN_STYLE = Automatic` with no `DEVELOPMENT_TEAM` and zero entitlements files exist anywhere in the repo. A Widget Extension requires an App Group entitlement shared between the main app and the extension so the extension can read the macro snapshot the main app writes; provisioning an App Group capability requires Xcode to register it against a real Apple Developer Team ID, which only resolves interactively in Xcode's "Signing & Capabilities" UI (selecting/creating a team, "Automatically manage signing" generating the App Group identifier and provisioning profiles) — there is no Apple ID/team signed into this environment and no `xcodebuild -allowProvisioningUpdates` credential path available, so this cannot be scripted blind. **Unblocking this requires a human, in Xcode GUI, on a machine with a signed-in Apple Developer account, to:** (1) select a Development Team for the `FoodDiary` target in Signing & Capabilities (currently none), (2) add a new "Widget Extension" target via File > New > Target (this generates the `.appex` product, `PBXNativeTarget`, build phases, and extension `Info.plist` correctly — far more reliable than hand-rolling via the `xcodeproj` gem), (3) add the "App Groups" capability to both the `FoodDiary` and the new widget extension targets, creating a shared group id (e.g. `group.com.motingo.fooddiary`), which auto-generates the `.entitlements` files for both targets. Once that scaffolding exists, the remaining work (a `WidgetSnapshotStore` writing today's macro totals to `UserDefaults(suiteName:)` after diary-affecting mutations, a `TimelineProvider` reading them back, and a `Widget`/`WidgetBundle` SwiftUI view reusing `MacroRing`) is straightforward and should follow the same TDD approach as `LogDiaryEntryService` for the snapshot read/write logic.
