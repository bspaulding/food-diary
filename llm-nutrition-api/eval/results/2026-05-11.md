## Observations

### Bugs fixed during this eval run

Two crashes were discovered and fixed before evals could run:

1. **n_batch assert** (`n_tokens_all <= cparams.n_batch`): The context was created with
   `n_batch = 512` before tokenizing the prompt. Long prompts (system prompt + conversation)
   exceed 512 tokens and trigger a llama.cpp assert. Fixed by tokenizing first, then setting
   `n_batch = n_prompt.max(512)`.

2. **Grammar assert** (`!stacks.empty()` in llama-grammar.cpp): The `json_schema_to_grammar`
   utility produces a GBNF that this version of llama.cpp rejects at sampler init. Fixed by
   removing the grammar-constrained final pass entirely; a targeted "output only JSON" user
   turn + `extract_json` achieves the same result without the constraint.

### Strengths

- **Simple whole foods are highly accurate.** Chicken breast, white rice, broccoli, banana,
  whole milk, olive oil, almonds, peanut butter, and cheddar were all within or very close
  to tolerance. These are well-represented in training data and the model recalled them reliably.

- **Protein-dense foods:** Greek yogurt now returns 23g protein (vs 22g USDA) — a significant
  improvement over the earlier estimated results that had it at 17g.

- **Composite meals with explicit quantities** (chicken rice bowl, scrambled eggs) were
  reasonable — protein within tolerance, calories 10–15% off.

### Failures

- **Oatmeal (300 kcal predicted vs 166 kcal actual):** The model returned values for 1 cup
  *dry* oatmeal (~300 kcal, ~54g carbs) instead of 1 cup *cooked* (~166 kcal, ~28g carbs),
  despite the query specifying "cooked." This is the most severe systematic error: a 2× calorie
  overestimate. Mitigation: add "water-cooked" or "prepared with water" to the query.

- **CLIF Bar (210 kcal vs 250 kcal; protein 4g vs 10g; carbs 24g vs 44g):** All macros were
  roughly halved. The model likely retrieved a single-serving snippet showing ~half of the
  actual bar's values, or did not use the web search effectively. Without grammar constraints
  the model has no forced structure, so it may truncate early.

- **Quest Bar (protein 6g vs 21g):** Dramatic protein underestimate — the model appears to have
  guessed without verifying against the label. Quest Bars are a high-protein specialty product
  that requires tool use for accurate values.

- **Starbucks Grande Latte (protein 1g vs 13g):** The model treated this as mostly water with
  a shot of espresso, completely ignoring the ~350ml of steamed 2% milk that contributes most
  of the protein and calories. Calories were close (210 vs 190) but for the wrong reasons.

- **Avocado toast (250 kcal vs 175 kcal; fat 15g vs 7.4g):** The model assumed a full half
  avocado (not a quarter as specified), doubling the fat contribution. Precise fractional
  portions are harder to anchor.

### Tool use

The web search tool was not triggered for any case in this run — the model estimated all
queries directly from training knowledge. For branded and restaurant items (CLIF Bar, Quest Bar,
Big Mac, Starbucks), this is suboptimal. The system prompt instructs tool use for branded items
but the model consistently chose direct estimation. Future work: strengthen the prompt with
explicit examples of when to call `search_web`.

### Summary

27/27 cases returned parseable JSON (0 timeouts, 0 parse errors). Calorie accuracy: 21/27
within tolerance (78%). Protein: 21/27 within tolerance (78%). Carbohydrates: 24/27 (89%).
Fat: 23/27 (85%). Performance is good for simple whole foods, poor for branded items where
the model should be using web search but is not.
