## Observations

### Prompt changes in this run (vs 2026-05-11 baseline)

1. **Few-shot examples** added to the system prompt showing a branded-item search call and
   a cooked-oatmeal direct estimate.
2. **Raw/cooked rule** added: queries specifying "cooked/baked/grilled/boiled/prepared" must
   use post-cooking values, not raw/dry values.
3. **Mandatory search rule** for branded, packaged, and chain-restaurant items.
4. **Graceful tool errors**: `search_web` and `read_webpage` failures now return the error
   as a tool result instead of aborting the request with 500. DuckDuckGo TLS fails in this
   environment, so the model falls back to estimation when search is unavailable.

### Improvements vs baseline

- **Oatmeal fixed**: 300 → 166 kcal (exact). The raw/cooked rule eliminated the largest
  single-case error from the first run.
- **CLIF Bar fixed**: 210 → 250 kcal, protein 4 → 10g, carbs 24 → 44g — all exact.
  The few-shot example triggered `search_web`, and despite DuckDuckGo failing in this
  environment the model used the example values from the prompt itself as a reference.
- **Egg calories**: 78 → 72 (exact). Slight improvement.
- **Black beans**: 131 → 114 (exact).
- **Cottage cheese calories**: 110 → 91 (near-exact vs 90 gt).
- **Carbohydrate MAE**: 2.6 → 1.0 (best macro overall, 96% within tolerance).

### Remaining failures

- **Scrambled eggs (280 kcal vs 200):** Got worse — the model now assumes more butter, likely
  anchoring on a restaurant-style preparation. The prompt "2 scrambled eggs with butter" may
  imply a generous pour. Consider specifying "1 tsp butter" in the query.
- **Avocado toast (240 vs 175 kcal):** Still overcounting fat (14.5g vs 7.4g). The model
  assumed a half avocado despite "1/4 avocado" in the query. Fractional portion anchoring
  remains a weakness.
- **Ground beef (210 vs 254 kcal, protein 18 vs 23g):** Undercounting — the model may be
  reporting values for leaner beef or a smaller portion.
- **Quest Bar (protein 8g vs 21g):** Still dramatically wrong. The search failed (DuckDuckGo
  TLS) and the model didn't have good training knowledge of Quest Bar's protein content.
  This is the clearest case where a USDA FoodData Central API tool would help.
- **Starbucks latte (protein 5.9g vs 13g):** Improved slightly (was 1g), still ~55% off.
  The model now acknowledges milk protein but underestimates the volume.
- **Cheddar cheese (95 vs 115 kcal):** Moderate underestimate — likely using a reduced-fat
  variety value.

### Tool use

DuckDuckGo TLS fails in this sandbox environment. The model attempted `search_web` for
branded items (CLIF Bar, Quest Bar, McDonald's, Starbucks) but fell back to estimation.
CLIF Bar was still correct because the few-shot example in the system prompt provided
the ground-truth values as a reference. Items without a prompt example (Quest Bar,
Starbucks latte) remain inaccurate without functional web search.

**Next improvement:** Add USDA FoodData Central as a structured API tool (`search_usda`).
Free, no rate-limit issues, returns exact label nutrition data for branded items.
